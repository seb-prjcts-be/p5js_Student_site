Debugging with AI

Traditional debugging uses console.log(), the error console and lots of patience. AI adds a new tool: you can present your code and ask what's going wrong.

That sounds simple, but the quality of the answer depends on how well you describe the problem. This is called prompting - the art of asking questions to AI.

Bad prompt vs. good prompt

Compare these two ways of asking for help:

Bad prompt

"My code doesn't work. Fix it."

This is too vague. The AI doesn't know what your code does, what's wrong, or what you expect. The answer will probably be vague too.

Good prompt

"I have a p5.js sketch that should draw a circle
that follows the mouse. But the circle stays in the
top-left corner. Here's my code:

function setup() {
  createCanvas(400, 300);
}

function draw() {
  background(220);
  ellipse(mouseX, mouseX, 50, 50);
}

What's going wrong?"

This is specific: you describe what you expect, what happens, and you provide the code. The AI can now look specifically - and will probably notice you used mouseX twice instead of mouseX, mouseY.

The prompt formula

An effective debugging prompt usually contains four parts:

  1. Context - What are you trying to make? ("A p5.js sketch with colliding particles")
  2. Expectation - What should happen? ("The particles should bounce off the edges")
  3. Problem - What actually happens? ("The particles disappear outside the canvas")
  4. Code - The relevant code, complete and copyable
Prompt formula:
"I'm making [context].
It should [expectation],
but instead [problem].
Here's my code: [code]"

Type it out, don't copy-paste

When the AI gives you a piece of code, it's tempting to just copy and paste it. Don't. Type the code out yourself, line by line.

Why? Typing it out forces you to read every line. You immediately notice which lines you don't understand - and then you can ask a targeted follow-up question. Copy-pasting skips that moment: you have working code, but no understanding. Next time something breaks, you're stuck again.

This rule matters most while you're still learning. Later, when you're fluent in p5.js, you can occasionally copy sections you fully understand (e.g. a repetitive block). In this course: always type it out.

Try this: ask AI for code for an effect you don't know yet (e.g. a mouse trail, a noise landscape). Type it out. Which lines do you understand right away? Which ones don't? Ask a follow-up question about the ones you don't.

Prompting for more than debugging

Good prompting is not only useful for bugs. You can use the same technique to:

  • Understand concepts - "Explain what map() does in p5.js, with a concrete example"
  • Explore alternatives - "Are there other ways to achieve this effect?"
  • Improve code - "How can I make this code more readable without changing its behaviour?"
  • Generate ideas - "Give me five ways to add interaction to this sketch"

Common prompting mistakes

  • Too vague - "Help me with my code" gives vague results
  • Too much context - Pasting a thousand lines of code without explanation overwhelms the model
  • No expectation - If you don't say what should happen, the AI can't know what's wrong
  • Blind trust - The AI can suggest a "fix" that moves the problem instead of solving it. Always test.
