Bugs horen bij programmeren - ze zijn niet het probleem, maar het zoekproces ernaar is de uitdaging. AI kan je helpen om fouten sneller te vinden, maar dan moet je wel de juiste vragen leren stellen. Welkom in de wereld van prompting.

Kernpunten

Debuggen met AI

Traditioneel debuggen doe je met console.log(), de foutconsole en veel geduld. AI voegt een nieuw gereedschap toe: je kunt je code voorleggen en vragen wat er misgaat.

Dat klinkt simpel, maar de kwaliteit van het antwoord hangt af van hoe goed jij het probleem beschrijft. Dit noemen we prompting - de kunst van het vragen stellen aan AI.

Slechte prompt vs. goede prompt

Vergelijk deze twee manieren om hulp te vragen:

Slechte prompt

"Mijn code werkt niet. Fix het."

Dit is te vaag. De AI weet niet wat je code doet, wat er mis is, of wat je verwacht. Het antwoord zal waarschijnlijk ook vaag zijn.

Goede prompt

"Ik heb een p5.js-sketch die een cirkel moet tekenen
die de muis volgt. Maar de cirkel blijft in de
linkerbovenhoek staan. Hier is mijn code:

function setup() {
  createCanvas(400, 300);
}

function draw() {
  background(220);
  ellipse(mouseX, mouseX, 50, 50);
}

Wat gaat er fout?"

Dit is specifiek: je beschrijft wat je verwacht, wat er gebeurt, en je geeft de code mee. De AI kan nu gericht kijken - en zal waarschijnlijk opmerken dat je twee keer mouseX gebruikt in plaats van mouseX, mouseY.

De prompt-formule

Een effectieve prompt voor debugging bevat meestal vier onderdelen:

  1. Context - Wat probeer je te maken? ("Een p5.js-sketch met deeltjes die botsen")
  2. Verwachting - Wat zou er moeten gebeuren? ("De deeltjes zouden moeten stuiteren tegen de randen")
  3. Probleem - Wat gebeurt er in werkelijkheid? ("De deeltjes verdwijnen buiten het canvas")
  4. Code - De relevante code, volledig en kopieerbaar
Prompt-formule:
"Ik maak [context].
Het zou moeten [verwachting],
maar in plaats daarvan [probleem].
Hier is mijn code: [code]"

Overtypen, niet kopiëren

Als de AI je een stuk code teruggeeft, is het verleidelijk om het gewoon te kopiëren en plakken. Doe dat niet. Type de code zelf over, regel voor regel.

Waarom? Overtypen dwingt je om elke regel te lezen. Je merkt onmiddellijk welke regels je niet begrijpt - en dán kun je gericht om uitleg vragen. Kopiëren slaat dat moment over: je hebt werkende code, maar geen begrip. Bij de volgende fout sta je weer met lege handen.

Deze regel geldt extra sterk zolang je aan het leren bent. Later, als je al vloeiend in p5.js werkt, mag je af en toe kopiëren voor stukken die je volledig snapt (bv. een repetitief blok). In deze cursus: altijd overtypen.

Probeer dit: vraag AI om code voor een effect dat je nog niet kent (bv. een muis-trail, een noise-landschap). Type het over. Welke regels begrijp je meteen? Welke niet? Stel over die laatste een vervolgvraag.

Oefening: vind de bugs

De sketch hieronder bevat bewuste fouten. Er zouden gekleurde cirkels moeten bewegen over het scherm, maar er gaat iets mis. Kun je de bugs vinden - eerst zelf, dan met AI?

Hints: Kijk goed naar de loop-conditie, de namen van de eigenschappen, en wat er gebeurt als een cirkel de rand bereikt. Er zitten drie bugs in.

Prompten voor meer dan debugging

Goede prompting is niet alleen nuttig voor bugs. Je kunt dezelfde techniek gebruiken om:

  • Concepten te begrijpen - "Leg uit wat map() doet in p5.js, met een concreet voorbeeld"
  • Alternatieven te verkennen - "Zijn er andere manieren om dit effect te bereiken?"
  • Code te verbeteren - "Hoe kan ik deze code leesbaarder maken zonder het gedrag te veranderen?"
  • Ideeën te genereren - "Geef me vijf manieren om interactie toe te voegen aan deze sketch"

Veelgemaakte prompt-fouten

  • Te vaag - "Help me met mijn code" geeft vage resultaten
  • Te veel context - Duizend regels code plakken zonder uitleg overweldigt het model
  • Geen verwachting - Als je niet zegt wat er zou moeten gebeuren, kan de AI niet weten wat er fout is
  • Blind vertrouwen - De AI kan een "fix" voorstellen die het probleem verplaatst in plaats van oplost. Test altijd.
